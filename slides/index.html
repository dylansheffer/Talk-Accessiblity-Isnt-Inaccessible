What is Accessibility?

"The inclusive practice of removing barriers that prevent interaction with, or access to websites, by people with disabilities.

When sites are correctly designed, developed and edited, all users have equal access to information and functionality."
Disabled World - Accessible Website Design Information

RealTalk

I generally like to keep my talks light, but before I continue we need to understand why this is such an important part of our job as developers.

It is critical we do this right

Accessibility isn't like keeping up with the latest development tools where it's nice to know, but can be shrugged away. This is about ensuring that all people have the same access to information regardless of ability. It is our shared responsibility that we preserve this fundamental human right to information and expression. And if this sounds overly dramatic listen this quote.
"Yes. [The Internet is] an essential tool. And, literally, a lifeline for many disabled people... Many disabled people have to spend long hours alone. Voice-activated computers are a means of communication that can prevent a sense of isolation."
- Christopher Reeve
For those who may not know, Christopher Reeve is best known for his role as Superman. Later in life, he was thrown from his horse and became quadriplegic due to his spinal cord injuries. If for only selfish reasons you ensure your work is accessible, remember, like Mr. Reeve, we are all an accident or a few years of age away from needing assistive technology.

  • Awareness
  • Context
  • Empathy

I'm not going to go into super detail about these topics, but I'll hit the main points.

WCAG

Web Content Accessibility Guidelines

The official guidelines published by the W3C on how to make the web accessible.

Web AIM's WCAG Checklist

The WCAG is a dense read and I think it's one of the reasons why developers are intimidated by accessibility. The good news is that the website webaim.org has created a more digestible checklist of the WCAG.

P.O.U.R

Perceivable

Operable

Understandable

Robust

Perceivable

Available to the senses either through the browser or through assistive technologies.

Mainly through vision and hearing. Assistive technologies include screen readers, screen enlargers, audio captions, etc.
Operable

Users can interact with all controls and interactive elements using either the mouse, keyboard, or an assistive device.

Understandable

Content is clear and limits confusion and ambiguity.

Robust

A wide range of technologies and devices can access the content.

Our sites need to be robust enough to be used in various contexts such as screen readers, screen enlargers, switches, etc. An easy to to ensure this is to give many ways to accomplish common tasks like page navigation and search.

A, AA, AAA, AAAHHHHHH!

The As are a levels of conformance to the WCAG

The way I think about the levels of conformance is. a- shitty manual transmission junker aa- typical sedan aaa - humvee
Level A

The very minimum needed, so that a website is usable for people with assistive technology


Junker car with a manual transmission

You can think a junker car with a manual transmission. It'll get you from point A to B, but it might be a bumpy experience especially if you don't already know how to use it.
Level AA

What is needed for a good user experience in most cases


A typical sedan

You can think a typical sedan. It's the most practical solution for most cases.
Level AAA

Very helpful in certain circumstances, but is either excessive or impractical in other cases.


A Humvee

A Humvee Useful when you need to drive over walls, but isn't always necessary for day to day use.

Section 508

Section 508 of the 1973 Rehabilitation Act


As of January 2018...

Section 508 == WCAG AA

The 1973 Rehabilitation Act layed a lot of the ground work of anti-discrimination legislation against people with disabilities. Section 508 used to be its own standard that you had to conform to, but as of January 2018, Section 508 and WCAG AA are the same!

How does assistive tech work with websites?

Accessibility Tree

A tree representation of the webpage that contains all meta-data about every page element.

What the browser does is when it parses an HTML document, it creates both the DOM and the Accessibility Tree.

What Assistive Tech Sees from the Tree

Name

State

Role

Value

You can inspect the a11y tree properties with Google's Accessibility Tools Plugin

How can I create a great A11y tree?

By utilizing HTML Semantic Elements!

<nav>

<button>

<section>

<h1>

Because HTML is accessible by default! Semantic HTML tags already have meaning associated with them.

Why HTML Semantics Matters

ARIA

Attributes used to modify the A11y Tree

Surgeon's Scalpel

What ARIA Does

  • Modifies the accessibility tree by adding semantics
  • Modifies existing semantics
  • Express UI Patterns
  • Add extra labels and descriptions
  • Establishing relationships between elements
  • Live Updates to a page

What ARIA Does Not Do

  • Modify an elementâ€™s appearance
  • Add focusability (tabbing)
  • Add keyboard event handling

Those are added in via CSS and Javascript

Easy A11y Wins

Use Semantic HTML Tags

They're accessible by default

A lot of work must be done to make elements from scratch

Button

We miss out on all the affordances (expected behavior) of a button

  • Styles for each button state
    (pressed, disabled, etc)
  • Not tabable (focusable)
  • Can't be triggered by synthetic clicks
    (spacebar, enter, etc)
  • Announced by a screen reader incorrectly

If we make a custom element, we must add back ALL expected functionality

Guideline Here

What about custom elements?

Start with a regular HTML element

Use ARIA tags to modify the difference

Example: Toggle Element

  1. Start with a button tag
  2. Add ARIA role of "switch"
  3. Give it the ARIA property of "aria-checked"
